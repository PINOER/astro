在构建独立站、小程序或 Web 应用时，网页的**首次加载速度（FCP / LCP）**对 SEO 排名和用户留存率有着举足轻重的影响。

打开浏览器的开发者工具（Network 面板），你会发现页面往往拉取了数十个仅有几 KB 大小的图标（Icon）、微型 Logo 或背景图案。**每个独立图片的请求都需要经历 DNS 解析、TCP 握手和 HTTP 报头传输**，这些累积起来的延迟往往比下载图片本身还要耗时。

为了解决这个问题，前端工程领域普遍采用了 **图片转 Base64 (DataURI)** 技术。

---

## ❓ 什么是图片的 Base64 / DataURI 编码？

简单来说，**Base64 是一种将任意二进制文件（如 PNG、JPG 或 SVG 图片）转换为一串 ASCII 文本字符的编码方式**。

通过在文本前加上 `data:image/png;base64,` 前缀，浏览器就可以直接识别并渲染这段字符串，而**不需要发起额外的网络请求**去服务器下载图片。

### 常见的嵌入形式：

* **HTML 中直接嵌入：**
  ```html
  <img src="data:image/png;base64,iVBORw0KGgoAAAANSUhEUgAA..." alt="Logo" />
  ```
* **CSS 背景图中使用：**
  ```css
  .icon {
    background-image: url("data:image/svg+xml;base64,PHN2Zy...");
  }
  ```

---

## ⚖️ 什么时候应该（以及不应该）使用 Base64？

虽然 Base64 可以减少 HTTP 请求数量，但它并不是盲目使用的“万能药”。理解它的利弊是做性能优化的关键：

### 建议使用的场景：
1. **微型图标与 Logo（小于 10 KB）：** 首屏关键路径上的图标（如购物车、搜索图标），转为 Base64 后随 HTML/CSS 一起秒级加载，彻底告别图片闪烁。
2. **CSS 关键背景纹理：** 直接写在 CSS 样式表中的微型图案或阴影渐变。
3. **单文件应用与邮件模板：** 无法依赖外部图片服务器、需要将图片完全内联的场景。

### 不建议使用的场景：
1. **大图与高清摄影图（大于 50 KB）：** 图片转为 Base64 后，**文件体积会增长约 33%**。大图转 Base64 会急剧膨胀 CSS 或 HTML 的体积，反而拖慢首次渲染。
2. **频繁变动的用户上传图片：** 无法利用浏览器的长期 HTTP 缓存机制。
